# Lore Agent vs Karpathy's LLM Wiki — Comparative Analysis

**Purpose**: This document compares the lore-agent project with Karpathy's LLM Wiki gist, identifies gaps, and tracks implementation status.

**Date**: 2026-04-10 (updated after external review)

---

## 0. Core Alignment: Compiled Knowledge

Both systems instantiate the same principle: **compile-time knowledge vs. query-time knowledge**. Instead of re-deriving answers from raw documents at every query, the LLM incrementally builds persistent knowledge artifacts that compound over time. This positions both within the lineage: Bush's Memex → Karpathy's LLM Wiki → Lore Agent as a programmatic instantiation with MCP protocol.

---

## 1. Systems Overview

### Lore Agent

A local knowledge retrieval and research agent with an MCP (Model Context Protocol) server interface. It allows AI coding assistants (Claude Code, Copilot) to query, save, and browse domain knowledge cards through structured tools.

**Architecture**:
- MCP server (`mcp_server.py`) exposing 4 tools: `query_knowledge`, `save_research`, `list_knowledge`, `capture_answer`
- BM25 + embedding hybrid search pipeline for retrieval
- Knowledge cards: markdown files with YAML frontmatter, organized by topic domain
- Research pipeline: web research → synthesize → distill → promote → reindex
- Lifecycle states: draft → reviewed → trusted → stale → deprecated
- 2 card types: **knowledge** (concepts, facts) and **method** (procedures, implementations)
- `[[card-id]]` wiki-links with bidirectional backlink resolution

**Key design decisions**:
- Cards are generated by AI during research, not manually authored
- Schema-validated structured answers (JSON Schema) are distilled into markdown cards
- Index is rebuilt after each card insertion for immediate retrievability
- Method cards include Expected Output and Concrete Example sections so AI can implement directly

### Karpathy's LLM Wiki

A personal wiki system where an LLM acts as a research assistant. The user asks questions, the LLM researches and writes wiki pages, and those pages persist across sessions.

**Architecture** (from the gist):
- Single persistent wiki file (`wiki.md`) or multi-file setup
- Manual prompt-driven workflow: user asks, LLM writes, file persists
- Cross-references via `[[wiki-links]]`
- Lint checks for content health (orphans, contradictions, staleness)
- Changelog in `log.md` tracking additions and changes
- Obsidian-compatible markdown for rich navigation
- Three-layer architecture: Raw Sources → Wiki (LLM-maintained) → Schema (conventions/rules)

## 2. Similarities

| Aspect | Lore Agent | LLM Wiki |
|--------|-----------|----------|
| Persistence | Markdown files with frontmatter | Markdown files |
| AI-driven content | Research → synthesis → card | Research → wiki page |
| Retrieval | BM25 + embedding hybrid search | Index scan + drill-down |
| Structure | Schema-validated JSON → markdown | Freeform markdown |
| Goal | AI-assisted domain knowledge | Personal knowledge wiki |

Both systems solve the same fundamental problem: **LLM context is ephemeral, but knowledge should persist**.

## 3. Differences

### Where Lore Agent is Stronger

1. **Structured schema validation**: Every answer passes through `answer.schema.json` validation before becoming a card. LLM Wiki has no schema enforcement.

2. **Programmatic tool interface**: MCP protocol lets any AI assistant query and save knowledge without manual prompting. LLM Wiki requires the user to explicitly say "write this to the wiki."

3. **Hybrid search pipeline**: BM25 + embedding retrieval with scoring gives ranked results. LLM Wiki uses index-scan-then-drill, which works at moderate scale (~100 sources) but has no relevance scoring.

4. **Lifecycle governance**: Card states (draft → reviewed → trusted → stale → deprecated) with transition validation and governance CLI. LLM Wiki has no formal lifecycle.

5. **Automatic domain classification**: Cards are auto-classified into domains and types based on query patterns. LLM Wiki depends on the author to categorize.

6. **Evidence traceability**: Each claim links to evidence IDs and source URLs. LLM Wiki doesn't enforce evidence chains.

### Where LLM Wiki is Stronger

1. **Explicit schema layer (Layer 3)**: Karpathy defines three layers — raw sources, the wiki, and a schema document telling the LLM how to maintain the wiki. Lore Agent's `answer.schema.json` partially fills this role but only covers answer structure, not knowledge base governance conventions.

2. **Typed edges**: Community implementations add semantic link types (`supports`, `contradicts`, `evolved_into`, `depends_on`). Lore Agent currently has untyped `[[card-id]]` links only.

3. **Progressive depth loading**: Community approaches use L0/L1/L2 frontmatter levels — the agent loads only the density it needs. Lore Agent loads full card content.

## 4. Gap Status (Updated Post-Implementation and Review)

### Gap 1: Card interconnection — IMPLEMENTED

`[[card-id]]` wiki-links are extracted during indexing (`extract_wiki_links()` in `common.py`). Bidirectional backlinks are computed and stored in the index. Links and backlinks appear in `query_knowledge` and `list_knowledge` results.

**Remaining**: Typed edges (`supports`, `contradicts`, etc.) and semantic graph queries.

### Gap 2: Content lint — IMPLEMENTED

`lint` command in `knowledge_governance.py` checks for:
- Orphan cards (no links in or out)
- Broken wiki-links (pointing to non-existent cards)
- Stale cards (not updated in N days)
- Schema drift (via existing `validate` command)

**Remaining**: Contradiction detection (requires embedding similarity + claim extraction + LLM judgment).

### Gap 3: Changelog — IMPLEMENTED

`append_changelog()` in `close_knowledge_loop.py` records card creation events with timestamps in `knowledge/changelog.md`.

**Remaining**: Track updates, deprecations, and lifecycle transitions (currently only creation is logged).

### Gap 4: Answer-writeback — IMPLEMENTED

`capture_answer` MCP tool accepts plain text Q&A, creates a draft card with low verification level, and triggers reindex. No structured JSON or evidence required.

## 5. Code-Level Fixes from Review

The external review identified several code-level issues. All fixed:

| Issue | Status |
|-------|--------|
| Incremental index doesn't remove deleted cards (ghost entries) | Fixed — only existing card paths are included |
| Manifest path inconsistency (mixed absolute/relative) | Fixed — canonical absolute paths used as keys |
| Query injected into LLM prompt without escaping | Fixed — `json.dumps()` escaping in `render_answer_bundle.py` |
| `except Exception: pass` silently swallows errors | Fixed — specific logging in `local_retrieve.py` and `orchestrate_research.py` |
| Orphan detection logic didn't properly account for backlinks | Fixed — simplified to check outgoing links and resolved backlinks |
